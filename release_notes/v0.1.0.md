# v0.1.0 Release Notes

**Released:** 2026-05-22
**Prior version:** none -- first release
**Tag:** v0.1.0
**Semver impact:** MINOR (initial public API; pre-1.0, callers should
expect MINOR bumps to break them)
**DOI:** [10.5281/zenodo.20350952](https://doi.org/10.5281/zenodo.20350952)

## Why v0.1.0

First citable release of `dcl_core`.  The package ships **two engines
under one import** -- the continuous-amplitude implementation ported
verbatim from Paper~I, and the new integer-token implementation that
enforces A=1 by integer arithmetic instead of float renormalisation.
Their coexistence is intentional: Paper~III's discrete-vs-continuous
probability comparison needs both side-by-side, and `dcl_core` is the
shared infrastructure both formalisms run against.

The public API in `src/dcl_core/__init__.py` (the submodule split:
`core` + `core3d`) is now stable enough for downstream papers to pin.
Pre-1.0 semver still applies -- minor bumps may break callers --
but the *shape* of the API (submodule split, factory pattern on
sessions, scheduler + residual contract) is unlikely to change
without a major bump.

## Public-API changes (vs no prior release)

### Added

**`dcl_core.core`** -- continuous-amplitude engine, port of
Paper~I's `src/core/`.  Public classes:

- `OctahedralLattice` -- bipartite lattice geometry, lattice
  basis vectors `RGB_VECTORS` / `CMY_VECTORS`.
- `CausalSession` -- continuous (`psi_R`, `psi_L`) Dirac spinor
  state with float-tolerance A=1 enforcement.
- `CompositeCausalSession` -- multi-session composite.
- `TickScheduler` -- tick orchestration + `ShuffleScheme`.
- `PhaseOscillator` -- phase-clock primitive.
- `UnityConstraint` -- `enforce_unity_spinor`, the post-hop
  renormalisation Paper~I relies on.

This submodule is verbatim Paper~I; downstream papers that pin to
this release get the Paper~I dynamics as their continuous baseline.

**`dcl_core.core3d`** -- integer-token engine.  Public classes /
factories:

- `BipartiteLattice` -- frozen `(shape, backend)` geometry, with
  `RGB_VECTORS` / `CMY_VECTORS` / `ALL_VECTORS` constants, a
  `parity_field()` method giving the per-site Z_2 grading, and
  `neighbour_offsets(parity)` returning the active sublattice
  vectors per tick.
- `DiscreteCausalSession` -- integer-token state (`N_R`, `N_L`,
  `phi_R`, `phi_L`).  Three construction paths:
  - **Bare constructor** allocates a vacuum (zeros + optional
    `momentum = (kx, ky, kz)` phase gradient).  Vacuum violates
    A=1 by design; callers populate before stepping.
  - `DiscreteCausalSession.delta_at(lattice, n_units, omega, position, momentum=...)`
    -- point source at a chosen site, split 50/50 between R / L
    (extra to R if `n_units` is odd).
  - `DiscreteCausalSession.wavepacket(lattice, n_units, omega, center, sigma, momentum=...)`
    -- minimum-uncertainty Gaussian wavepacket of width
    `sigma`; the corresponding `Δp = 1 / (2 σ_x)` per axis.  IS
    Paper~III's min-Δp generator.
  - `DiscreteCausalSession.from_arrays(lattice, n_units, omega, N_R, N_L, phi_R=None, phi_L=None)`
    -- arbitrary initial state, validates that the array sum
    equals `n_units`.
- `BresenhamResidual` -- floor + global top-up rebalance with a
  persistent fractional carry.  The carry is **real** in v0.1.0;
  the parked hypothesis that it should be complex (representing
  virtual-particle amplitude) is documented in
  `notes/bresenham_residual_design.md` and the determination is
  expected to live downstream in Paper~III's min-Δp experiments.
- `HopOperator` -- bipartite Dirac evolution.  `step` returns
  the analytical complex (`psi_R_new`, `psi_L_new`);
  `fourier_kernel(k)` returns the small-k structure factor
  `(S_RGB - S_CMY) / 2`.
- `TickScheduler` -- per-tick loop wiring hop -> renorm ->
  quantise -> writeback -> `assert_unity`.  Supports multiple
  independent sessions (pairwise interactions deferred to
  v0.2.0).  An `on_tick_complete: Callable[[TickScheduler], None]`
  hook fires after each tick for instrumentation / observable
  extraction.
- `dcl_core.core3d.clifford` (opt-in submodule, requires sympy) --
  `gamma_0..gamma_3`, `ETA_MINKOWSKI`, `GAMMAS` tuple in the
  Dirac representation, mostly-minus convention.  Sympy is
  lazy-imported so the module is safe to load without the
  `[sympy]` extra installed.

**Backends.**

- `dcl_core.core3d.backends.cpu` -- NumPy reference implementation.
  Primitives: `zeros`, `zeros_int`, `indices`, `shift`,
  `sum_all`, `sqrt`, `exp`, `floor`, `cos`, `sin`.
- `dcl_core.core3d.backends.gpu` -- CuPy stubs.  All allocators
  work; the hop's `RawKernel` bodies are TODO strings (deferred
  to v0.2.0).

## New experiments

1. **`exp_01_throughput`** (ACTIVE) -- Tick-throughput sweep
   across lattice shapes `8^3` ... `256^3`, writes
   `data/exp_01_throughput.{npy,json}` for the throughput
   baseline.  Designed to be re-run on upgraded hardware
   and diffed.
2. **`exp_02_phase_profile`** (ACTIVE) -- cProfile breakdown of
   a single tick at fixed shape (default `64^3`), output goes
   to `data/exp_02_phase_profile.{prof,log}`.  Identifies
   memory-bandwidth vs per-element-compute vs Python-overhead
   bottlenecks.
3. **`exp_00_hop_drift`** (STUB carried over from the template).
   Designed but not implemented; un-stub when needed to
   characterise pre-rounding drift on the continuous core.

## Performance notes

The engine is CPU-only in v0.1.0; the CuPy GPU backend has
allocators but no kernels.  On a 2018-era Ryzen 7 2700 + DDR4-2400
+ NumPy 2.x, characteristic per-tick wall-clock times are:

| Shape    | Sites      | Per-tick   |
|----------|-----------:|-----------:|
| `8^3`    | 512        | ~325 µs    |
| (larger) | (more)     | re-run `exp_01_throughput` on the target hardware |

Run `python experiments/exp_01_throughput.py` for the full sweep
and `data/exp_01_throughput.{npy,json}` for the structured output.

The hot path is dominated by memory-bandwidth-bound primitives
(`numpy.roll` in the hop, `numpy.abs` / `numpy.angle` in the
writeback), so the v0.2.0 GPU port is expected to give the largest
speedup at `64^3` and above.

## Reproducibility

```sh
git clone https://github.com/JackDMenendez/dcl-core.git
cd dcl-core
git checkout v0.1.0
python -m venv .venv
source .venv/bin/activate   # POSIX
# .venv\Scripts\activate.bat # Windows cmd
pip install -e ".[dev]"
pytest tests/ -v
python experiments/exp_01_throughput.py
```

Expect ~5 seconds for the test suite and ~5-15 minutes for the
throughput sweep on a 2018-class CPU.  Bumping `N_MEASURE` in
`exp_01_throughput.py` by ~10x gives an overnight-quality baseline
(~30-60 minutes).

## What is NOT in v0.1.0

Documented in CLAUDE.md (`CURRENT STATUS` -> "What is NOT in
scope for v0.1.0"):

- **GPU backend kernels.**  The `cupy` allocators are wired but
  `RawKernel` bodies in `backends/gpu.py` are TODO strings.
  Deferred to v0.2.0.
- **Pairwise multi-session interactions.**  The scheduler's per-
  tick loop has a hook point between the hop and quantise steps
  for Coulomb / gauge / emission coupling; nothing is wired
  through it in v0.1.0.  Deferred to v0.2.0.
- **Complex `BresenhamResidual.carry`.**  Real carry only.  The
  decision to switch to complex is expected to be made
  downstream by Paper~III's min-Δp experiments; see
  `notes/bresenham_residual_design.md`.
- **Gaussian / orbital-style factories beyond `wavepacket`.**
  Use `from_arrays` for arbitrary initial states until a
  specific factory (`orbital_1s`, etc.) is justified by a
  downstream experiment.
